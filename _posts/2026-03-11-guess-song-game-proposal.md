## 一、專案背景

本遊戲伺服器（XPG Server）目前已運行「火箭遊戲（Rocket）」與「骰寶（XocDia）」兩款遊戲。為擴充平台多元性、提升使用者黏著度，提案新增**猜歌遊戲（GuessSong）**作為第三款遊戲內容。

猜歌遊戲定位為**社交互動型益智遊戲**，門檻低、趣味性高，適合聚會、團隊活動、品牌活動等場景，與現有博弈類遊戲形成互補，觸及更廣泛的使用者族群。

---

## 二、遊戲概念

播放一段 **30 秒音樂片段**，玩家在時間限制內，透過**點擊字元**的方式拼出歌名，搶先答對即可得分。

### 核心體驗

- 聽到熟悉旋律的驚喜感
- 快速點擊搶答的緊張感
- 與隊友合作、超越對手的競爭感

---

## 三、遊戲模式

### 3.1 個人模式

| 項目 | 說明 |
|------|------|
| 人數 | 1 人 |
| 進入方式 | 直接進入，無需等待 |
| 計分 | 個人歷史紀錄留存 |
| 適合場景 | 碎片時間、練習、排行榜挑戰 |

### 3.2 多人模式

| 項目 | 說明 |
|------|------|
| 人數 | 2 ～ N 人 |
| 進入方式 | 輸入 6 碼房間代碼加入 |
| 計分 | 個人積分排行 |
| 適合場景 | 朋友同樂、小型競賽 |

### 3.3 隊伍模式

| 項目 | 說明 |
|------|------|
| 人數 | 最多 100 人 |
| 進入方式 | 輸入代碼 → 輸入暱稱 → 選擇或被分配隊伍 |
| 隊伍數量 | 2 ～ 5 隊，隊伍名稱可自訂（如隊長名字）|
| 人員分配 | 自由選隊 或 系統平均分配 |
| 計分 | 個人得分累計至隊伍，以隊伍總分決勝負 |
| 適合場景 | 企業年會、班級活動、大型聚會、品牌活動 |

#### 隊伍平均分配邏輯

設定 3 隊、10 人加入，系統自動平均分配結果為 **4、3、3**，依序將玩家分配至人數最少的隊伍，同人數時隨機分配，確保公平性。

## 四、作答機制

### 字元點擊設計

考量行動裝置輸入不便，採用**字元點擊**方式作答：

1. Server 推送題目字元列表（含答案字元 + 隨機干擾字元，共約 12 ～ 24 字）
2. 玩家點擊字元依序拼出歌名
3. 點擊已選字元可取消，重新選擇
4. 拼完後送出，或時間到自動結算

```
┌─ 已選 ──────────────────┐
│  [小] [_] [_]           │
└─────────────────────────┘

┌─ 選字區（隨機排列）──────────────────┐
│  [愛] [幸] [你] [運] [我]           │
│  [情] [小] [天] [的] [心]           │
│  [夜] [想] [花] [給] [月]           │
└─────────────────────────────────────┘
```

> 資料來源控制為**純中文歌名**，避免英文或數字造成的操作困難。

## 五、計分方式

創房時可選擇計分模式：

| 模式 | 說明 |
|------|------|
| 答對加分 | 答對固定得 100 分，輕鬆模式 |
| 速度加分 | 越早作答分數越高（滿分 100，每秒遞減）|
| 兩者疊加 | 答對才計時間分，競技模式 |

## 六、歌曲題庫

### 資料來源

使用 **iTunes 官方公開 Search API**（免費、合法、穩定）：

```
https://itunes.apple.com/search?term=周杰倫&media=music&limit=50
```

API 提供欄位：歌名、歌手、年份、**30 秒試聽連結**、封面圖片

### 資料庫架構

```
songs
├── id
├── track_name      歌名
├── artist_name     歌手
├── release_year    年份
├── preview_url     30 秒試聽連結（Apple 授權）
├── artwork_url     封面圖
├── source          "itunes" / "user_upload"
├── is_active       是否上架
└── created_at

song_tags（擴充分類）
├── song_id
└── tag             "華語"、"西洋"、"90年代"、"KTV金曲" 等
```

### 擴充性

- 未來新增歌曲**只需異動資料庫**，Server 不需重啟、程式不需修改
- 預留**使用者上架**機制，可提交歌曲申請，通過審核後加入題庫

## 七、技術架構

### 與現有系統整合

本遊戲完全遵循現有 XPG Server 的 **Content 擴充架構**，新增一個 `Content/GuessSong/` 模組即可，不影響現有遊戲運作。

```
GameServer/
└── Content/
    ├── Rocket/      ← 現有
    ├── XocDia/      ← 現有
    └── GuessSong/   ← 新增
        ├── _GameStruct.cs
        ├── GuessSong.cs
        ├── GuessSong.Stage.cs
        ├── GuessSong.Command.cs
        ├── GuessSongLobbyAdapter.cs
        ├── GuessSongLobbyAdapter.BgLine.cs
        └── GuessSongLobbyAdapter.Report.cs
```

### 遊戲場景流程

```
等待（Waiting）
  → 開始一輪（RoundStart）：推送音樂 + 字元列表
  → 開放作答（AnswerOpen）：倒數計時
  → 截止作答（AnswerClose）：時間到或全員作答
  → 公布結果（ShowResult）：答案、本輪得分、隊伍排行
  → 下一輪 / 遊戲結束（GameEnd）：最終結算
```

### 通訊協定（WebSocket JSON）

| 命令 | 說明 |
|------|------|
| `gscr` | 創建猜歌房 |
| `gsjr` | 以代碼加入房間 |
| `gsgi` | 房間完整資訊 |
| `gsts` | 等待室隊伍人數更新 |
| `gsq`  | 推送題目（字元列表）|
| `gsa`  | 玩家提交答案 |
| `gsrr` | 本輪結果 |
| `gsfr` | 最終結算 |

### 開發期加速策略

開發初期使用 **mockJSON** 假資料模擬題庫 API，待遊戲邏輯穩定後再接入真實 iTunes API，無需等待後端就緒即可平行推進。

## 九、獲利模式

### 階段一：免費開放（導入期）

目標：累積使用者基礎、驗證市場需求、建立口碑。

| 策略 | 說明 |
|------|------|
| 全功能免費 | 個人模式、多人模式、隊伍模式皆免費使用 |
| 基本題庫開放 | 提供數百首熱門中文歌曲 |
| 廣告置入 | 等待室、結算畫面置入橫幅廣告（非強制影片）|
| 資料積累 | 收集用戶行為資料，作為後續付費功能設計依據 |

### 階段二：Freemium 模式（成長期）

| 功能 | 免費版 | 付費版 |
|------|--------|--------|
| 個人模式 | ✅ | ✅ |
| 多人模式 | ✅（最多 10 人）| ✅（無限）|
| 隊伍模式 | ✅（最多 20 人）| ✅（最多 100 人）|
| 題庫類型 | 熱門流行 | + 年代、地區、語言等分類 |
| 房間時效 | 2 小時 | 24 小時 |
| 自訂題庫 | ❌ | ✅ |
| 歷史紀錄 | 最近 10 局 | 完整紀錄 |
| 廣告 | 有 | 無 |

### 階段三：B2B 企業方案（變現期）

| 方案 | 內容 | 適合客戶 |
|------|------|---------|
| **活動方案** | 單次使用，最多 100 人，指定歌單，含活動主題頁 | 企業年會、尾牙、品牌活動 |
| **品牌方案** | 嵌入品牌 LOGO、自訂房間主題色、活動後提供參與報表 | 品牌行銷活動 |
| **教育方案** | 教師建立班級房間，學生以代碼加入，可上傳自訂曲目（如英文歌填空）| 學校、補習班 |
| **平台授權** | 將猜歌遊戲模組授權給其他平台整合 | 娛樂 App、KTV 系統 |

### 階段四：使用者上架機制（生態期）

| 機制 | 說明 |
|------|------|
| 免費上架 | 使用者可提交歌曲，審核後進入公共題庫 |
| 創作者分潤 | 若題庫採用付費分類，上架歌曲的提交者獲得積分回饋 |
| 私人題庫 | 付費用戶可建立只有自己房間使用的私人歌單 |

## 十、風險評估

| 風險 | 等級 | 因應方式 |
|------|------|---------|
| iTunes API 異動或停用 | 低 | 預留多資料來源介面，可切換 Spotify、YouTube Music API |
| 音樂版權爭議 | 低 | 使用 iTunes 官方授權試聽連結，不自行儲存音樂檔案 |
| 大人數（100人）效能 | 中 | 廣播採聚合推送，排行榜不顯示全部名單，已有設計規避 |
| 使用者上架違規內容 | 中 | 初期人工審核，後期加入關鍵字過濾機制 |

## 十二、結語

猜歌遊戲具備以下核心優勢：

1. **低開發風險**：完全沿用現有架構，擴充方式已有前例（Rocket、XocDia）
2. **高社交傳播性**：房間代碼分享機制天然帶有病毒式傳播特性
3. **多場景適用**：個人、朋友、企業活動皆可使用，受眾廣泛
4. **清晰的商業路徑**：從免費導入到企業 B2B，有明確的升級路線
5. **內容擴充性強**：歌庫與遊戲邏輯解耦，未來維護成本極低

建議以「**先免費累積用戶、再精準付費轉化**」的策略推進，初期最快 6 週即可完成可上線版本。
